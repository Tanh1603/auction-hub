# 3.4.10 Reject Registration

## 1. Use Case Description

| Field              | Description                                                                                                             |
| ------------------ | ----------------------------------------------------------------------------------------------------------------------- |
| **Name**           | Reject Registration                                                                                                     |
| **Description**    | This use case allows the Admin to update existing Registration information in the system.                               |
| **Actor**          | Admin                                                                                                                   |
| **Trigger**        | When the Admin clicks on the 'Reject' button on the RegistrationApprovalPage.                                           |
| **Pre-condition**  | • Admin's device must be connected to the internet.<br>• Admin is signed in with their account.                         |
| **Post-condition** | The Registration information will be updated in the system and display new record on RegistrationApprovalPage datagrid. |

## 2. Sequence Flow (MVC)

```plantuml
@startuml
autonumber
skinparam sequenceMessageAlign center

actor "Admin" as User
boundary "RegistrationApprovalPage" as View
control "RegistrationController" as Controller
control "RegistrationService" as Service
entity "ParticipantRepository" as ParticipantRepo

User -> View: Select Registration -> Reject (Reason)
View -> Controller: POST /register-to-bid/admin/reject\n{ auctionId, userId, reason }
activate Controller

Controller -> Service: rejectRegistration(auctionId, userId, reason)
activate Service

Service -> ParticipantRepo: findParticipant(auctionId, userId)
activate ParticipantRepo
ParticipantRepo --> Service: ParticipantEntity
deactivate ParticipantRepo

alt Participant Not Found
    Service --> Controller: Throw NotFoundException
    Controller --> View: 404 Not Found
    View --> User: Display Error Message
else Found
    ' FIX: Combined message onto one line; used \n for the visual line break
    Service -> ParticipantRepo: updateParticipant({ status: 'REJECTED', \nrejectionReason: reason })
    activate ParticipantRepo
    ParticipantRepo --> Service: UpdatedParticipant
    deactivate ParticipantRepo

    Service --> Controller: SuccessResponse
    Controller --> View: 200 OK (Rejected)
    View --> User: Display Success Confirmation
end

deactivate Service
deactivate Controller
@enduml
```

## 4. Business Rules

| Activity      | BR Code   | Description                                                                                                                                                                                                                                                                                                                             |
| :------------ | :-------- | :-------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **(1)**       | **BR-01** | **Displaying Rule (Registration Approval Page):**<br>When Admin reviews a registration, system displays `RegistrationApprovalPage`.<br>System displays registration details, bidder info, and submitted documents.<br>System displays "Reject" button with reason input field.                                                          |
| **(2)-(4)**   | **BR-02** | **Validation Rule (Input - Front-end):**<br>When Admin enters rejection reason, system uses `Text_change()` method.<br>System enables "Submit Rejection" button when reason is provided.<br>On submit, system displays MSG 11 ("Are you sure you want to reject this registration?").                                                   |
| **(5)**       | **BR-03** | **Validation Rule (Authorization - Back-end):**<br>System checks if requestor role is `admin` or `auctioneer` in `USERS` table.<br>If unauthorized:<br>$\rightarrow$ System displays MSG 5 ("Forbidden") on the View.                                                                                                                   |
| **(7)-(8.1)** | **BR-04** | **Validation Rule (Record Existence - Back-end):**<br>System retrieves data from the 'AUCTION_PARTICIPANT' table in the database (Refer to 'AUCTION_PARTICIPANT' table in 'DB Sheet' file) based on the auction ID and user ID.<br>If record not found:<br>$\rightarrow$ System displays MSG 20 ("Registration not found") on the View. |
| **(9)-(10)**  | **BR-05** | **Storing Rule:**<br>System saves to `AUCTION_PARTICIPANT` table:<br>- `status = 'REJECTED'`<br>- `rejectionReason = [input reason]`<br>- `rejectedAt = now()`                                                                                                                                                                          |
| **(11)**      | **BR-06** | **Processing Rule (Notification):**<br>System triggers email notification to bidder with rejection reason.<br>Notification explains why registration was denied.                                                                                                                                                                        |
| **(13)**      | **BR-07** | **Displaying Rule (Success Confirmation):**<br>System displays MSG 7 ("Registration rejected successfully") on the View.<br>System refreshes registration list showing updated `REJECTED` status.                                                                                                                                       |
